# 文档智能分析系统 — 答辩文档

## 一、项目概述

### 1.1 项目背景

在大数据时代，企业和个人面临海量文档数据（PDF 报告、Word 文档、PPT 演示、Excel 表格、扫描件、网页、技术手册等），传统的关键词搜索无法满足语义级的深度检索需求。本项目构建了一个基于 **RAG（检索增强生成）** 的文档智能分析系统，用户上传任意常见文档后，系统自动将非结构化文档转化为结构化 JSON 并建立向量知识库，支持基于文档内容的自然语言问答。

### 1.2 核心功能

| 功能 | 描述 |
|------|------|
| 多格式文档上传 | 支持 PDF(含扫描版)/Word(.doc/.docx)/PPT/Excel/HTML/图片/文本等 **5 大类格式** |
| 智能文本提取 | 根据文件类型自动选择加载器；PDF 三级降级策略(文本→PyMuPDF→OCR) |
| 扫描件 OCR | 扫描版 PDF 和图片通过 DashScope 多模态 API (qwen-vl-max) 识别文字 |
| JSON 结构化 | 智能分块(chunk)，生成带完整元数据的结构化 JSON |
| 向量知识库构建 | DashScope 嵌入 + Chroma 持久化存储，支持增量追加 |
| AI 智能问答 | 基于 RAG 的语义检索 + DeepSeek-V3 生成，SSE 流式返回 |
| **知识库持久化** | **重启后自动恢复文件列表，无需重新上传，向量数据持久化在 chroma_db/** |
| 安全防护 | 基础安全配置(限流/认证/CSP/XSS/文件净化/输入校验/日志脱敏/访问控制) |

## 三、核心模块详解

### 3.1 文档加载模块 (`document_loader.py`) — 5大类格式 + OCR

**设计模式:** 策略模式 (Loader Registry)

**支持格式一览:**

| 类别 | 格式 | 加载器 | 说明 |
|------|------|--------|------|
| 文档 | `.pdf` | `load_pdf` | 三级降级：PyPDF → PyMuPDF → API OCR |
| 文档 | `.doc` | `load_doc` | 旧版 Word，先试 python-docx → 提示转 .docx |
| 文档 | `.docx` | `load_docx` | python-docx 段落提取 |
| 演示 | `.pptx` | `load_pptx` | python-pptx 幻灯片文本 |
| 表格 | `.xlsx` | `load_xlsx` | openpyxl 工作表数据 |
| 网页 | `.html` `.htm` | `load_html` | BeautifulSoup 去标签提取 |
| 图片 | `.png` `.jpg` `.jpeg` `.bmp` `.tiff` `.tif` | `load_image` | API OCR 识别 |
| 文本 | `.txt` `.md` `.csv` `.json` `.xml` `.log` | `load_text` | UTF-8 直接读取 |

**PDF 三级降级 OCR 策略（核心创新）:**

```
PDF 文件上传
    │
    ├── 第1步: PyPDF 提取文本层
    │   └── 有内容 ✓ → 直接返回 (最快)
    │
    ├── 第2步: PyMuPDF (fitz) 提取
    │   └── 有内容 ✓ → 返回 (兼容更多 PDF 变体)
    │
    └── 第3步: 逐页渲染为图片 → DashScope qwen-vl-max 多模态 API OCR
        └── 日志: "OCR 处理第 X/N 页..."
        └── 有内容 ✓ → 返回 (扫描版/图片型 PDF)
        └── 无内容 ✗ → 提示"文档可能为纯图片或手写内容"
```

**OCR 实现 — 云端多模态方案（避免本地依赖冲突）:**

```python
def _ocr_image(image_bytes: bytes) -> str:
    """使用 DashScope 多模态模型 (qwen-vl-max) 进行 OCR"""
    client = OpenAI(api_key=DASHSCOPE_API_KEY, base_url=BASE_URL)
    img_b64 = base64.b64encode(image_bytes).decode("utf-8")

    response = client.chat.completions.create(
        model="qwen-vl-max",
        messages=[{
            "role": "user",
            "content": [
                {"type": "image_url", "image_url": {"url": f"data:image/png;base64,{img_b64}"}},
                {"type": "text", "text": "请完整提取并输出这张图片中的所有文字内容..."},
            ],
        }],
        temperature=0.1,
        max_tokens=4096,
    )
    return response.choices[0].message.content
```

**选择 API OCR 而非本地 OCR 的原因:**
- Python 3.13 + Windows 环境下，PaddleOCR/EasyOCR 存在 NumPy 版本冲突和网络下载障碍
- DashScope 多模态 API 国内可访问，中文识别准确率高
- 零本地模型文件（~2GB），部署轻量
- PyMuPDF `try/finally` 确保文件句柄释放，解决 Windows 文件占用问题

**Loader 注册表关键设计:**
- `LOADERS` 字典是格式注册的**唯一权威源**，扩展名验证由此派生
- 新增格式只需添加一个 key-value 对，符合开闭原则
- 元数据（filename, file_type）由 `load_file` 统一注入，各 loader 不重复设置

### 3.2 文本处理模块 (`text_processor.py`)

**核心流程:**

```
原始文档 → RecursiveCharacterTextSplitter → 结构化 JSON
            chunk_size=1500, overlap=300
```

**JSON 输出结构:**

```json
[{
  "chunk_id": 0,
  "content": "人工智能(AI)是计算机科学的一个重要分支...",
  "char_count": 188,
  "metadata": {
    "source": "D:\\...\\uploads\\abc123_test.txt",
    "file_type": ".txt",
    "filename": "abc123_test.txt",
    "page": 0
  }
}]
```

**设计亮点:**
- 使用 `@dataclass(frozen=True)` 的 `ProcessResult` 替代裸 dict，类型安全 + IDE 自动补全
- `chunk_overlap=300` 保证相邻 chunk 间有足够的上下文重叠，防止关键信息在分块边界丢失
- `docs_to_json()` 使用列表推导式，简洁高效

### 3.3 向量存储模块 (`vector_store.py`)

```
文本 Chunk → DashScopeEmbeddings(text-embedding-v2) → 1024维向量 → ChromaDB (持久化)
```

**关键设计:**
- `@lru_cache(maxsize=1)` 缓存嵌入模型实例，避免每次操作重建 HTTP 连接
- 持久化到 `chroma_db/` 目录，服务重启后无需重新处理文档
- 支持增量添加：`add_documents()` 向已有知识库追加新文档，无需重建

### 3.4 RAG 问答链 (`qa_chain.py`)

**LangChain LCEL 声明式链式调用:**

```python
def create_rag_chain(retriever, *, streaming=False):
    model = ChatOpenAI(
        api_key=DASHSCOPE_API_KEY,
        base_url=BASE_URL,
        model=LLM_MODEL,
        streaming=streaming,       # 唯一差异点 — 通过参数控制
    )
    prompt = ChatPromptTemplate.from_messages([
        ("system", SYSTEM_PROMPT),
        ("human", "{input}"),
    ])
    return (
        {"context": retriever | format_docs, "input": RunnablePassthrough()}
        | prompt
        | model
        | StrOutputParser()
    )
```

**设计要点:**
- 原本 `create_qa_chain` 和 `create_streaming_chain` 两个函数 80% 重复 → 合并为一个，通过 `streaming` 参数区分
- 明确指示 LLM "找不到就说不知道"，防止幻觉
- `RunnablePassthrough()` 是 LCEL 规范用法，直接透传用户输入

**Prompt 模板:**

```
System: 你是一个基于文档内容进行问答的助手。
        请使用以下检索到的文档内容来回答用户的问题。
        如果你无法从文档中找到答案，请诚实地说明你不知道，
        不要编造信息。请尽量用八句话以内简洁地回答。

        文档内容:
        {context}

Human: {input}
```

**性能优化:**
- **Chain 缓存**: `_get_chain()` 复用 LLM HTTP 连接池，文件变更时 `_invalidate_cache()` 自动重建
- **嵌入缓存**: `@lru_cache` 避免重复创建 DashScope 连接
- **空 chunk 守卫**: `chunk_count == 0` 时提前拦截，防止崩溃
- **文件句柄保护**: `try/finally` 确保 PyMuPDF 释放，`OSError` 捕获防止清理失败
- **文件名净化**: `sanitize_filename()` 处理 Unicode/特殊字符/长度限制
- **知识库状态持久化**: 上传/删除操作自动保存到 `data/knowledge_state.json`，启动时恢复

### 3.5.1 知识库持久化机制

系统重启后无需重新上传文件，所有状态自动恢复：

```
启动 python app.py
    │
    ├── 1. load_vectorstore()
    │       └── 加载 chroma_db/ → 向量数据恢复
    │
    └── 2. _load_knowledge_state()
            ├── 读取 data/knowledge_state.json
            ├── 逐条验证:
            │   ├── data/uploads/{id}_{name} 存在? + data/json/{name}.json 存在?
            │   │   ├── 都存在 → 恢复到 processed_files
            │   │   └── 缺失 → 跳过，记录日志
            └── 前端文件列表恢复正常显示
```

**持久化数据层:**

| 数据 | 存储位置 | 重启后 |
|------|---------|--------|
| 原始文件 | `data/uploads/` | 保留 |
| JSON 结构化 | `data/json/` | 保留 |
| 文件列表状态 | `data/knowledge_state.json` | 读取恢复 |
| 向量嵌入 | `chroma_db/` | 自动加载 |

**保存/加载函数:**

```python
def _save_knowledge_state():
    """上传/删除后自动调用，将 processed_files 写入 JSON"""
    with open(KNOWLEDGE_STATE_FILE, "w") as f:
        json.dump(list(processed_files.values()), f, ensure_ascii=False, indent=2)

def _load_knowledge_state():
    """启动时调用，验证磁盘文件后恢复内存状态"""
    records = json.load(open(KNOWLEDGE_STATE_FILE))
    for rec in records:
        if upload_path.exists() and json_path.exists():
            valid_records[rec["id"]] = rec  # 恢复
        else:
            logger.info(f"跳过已丢失的文件: {rec['filename']}")  # 自动跳过
```

## 四、数据流详解

### 4.1 文档上传处理流

```
用户选择/拖拽文件
    │
    ▼
前端验证 → 扩展名白名单
    │
    ▼
Flask 接收 → 文件大小检查 (32MB MAX_CONTENT_LENGTH)
    │         → IP 频率限制 (10次/小时)
    │         → 可选 Bearer Token 认证
    ▼
文件名净化 → sanitize_filename()
    │         Unicode 规范化 + 控制字符移除 + 特殊字符替换 + 200字符截断
    ▼
保存到 data/uploads/{file_id}_{sanitized_name}
    │
    ▼
document_loader.load_file()
    ├── .pdf  → PyPDF → PyMuPDF → API OCR (三级降级)
    ├── .doc  → python-docx → 提示转 .docx
    ├── .docx → python-docx 段落提取
    ├── .pptx → python-pptx 幻灯片文本
    ├── .xlsx → openpyxl 工作表数据
    ├── .html/.htm → BeautifulSoup 去标签
    ├── .png/.jpg/... → API OCR 识别
    └── .txt/.md/.csv/... → 直接读取
    │
    ▼
text_processor.process_documents()
    ├── RecursiveCharacterTextSplitter(chunk_size=1500, overlap=300)
    ├── docs_to_json() → 结构化 JSON (chunk_id + content + char_count + metadata)
    └── save_json() → data/json/{filename}.json
    │
    ▼
空内容检查 → chunk_count == 0? → 提示"文档内容为空或扫描件无法识别"
    │
    ▼
vector_store.create_vectorstore() 或 add_documents()
    ├── DashScopeEmbeddings(text-embedding-v2) → 1024维向量
    └── Chroma.from_documents() → 持久化到 chroma_db/
    │
    ▼
_invalidate_cache() → 清除缓存的 RAG chain (下次请求重建)
    │
    ▼
_save_knowledge_state() → 文件列表持久化到 data/knowledge_state.json
    │
    ▼
返回 JSON: {success, file_id, filename, chunks, total_chars, json_path}
```

### 4.2 问答检索流

```
用户输入问题
    │
    ▼
Flask 接收 → 非空检查 → 长度检查 (≤2000字符)
    │         → IP 频率限制 (30次/分钟)
    │         → 知识库就绪检查 (vectorstore is not None)
    ▼
_get_chain() → 复用缓存 chain 或新建 (首次/失效后)
    │
    ▼
retriever.invoke(question)
    ├── 问题向量化 → text-embedding-v2 → 1024维
    ├── Chroma 余弦相似度检索 → Top-K (K=4)
    └── 返回 4 个最相关文档块
    │
    ▼
format_docs() → "\n\n".join(doc.page_content for doc in docs)
    │
    ▼
Prompt 拼接 → System Prompt 填入 {context} + Human Message 填入 {input}
    │
    ▼
ChatOpenAI(deepseek-v3.2-exp, streaming=True)
    │
    ▼
LCEL .stream(question) → 逐 token 产出
    │
    ▼
SSE 格式化 → "data: {json}\n\n" → Flask Response (text/event-stream)
    │
    ▼
前端 parseSSEStream() → contentEl.textContent += chunk → 逐字渲染
```

## 五、安全设计

### 5.1 基础安全配置

| 层级 | 措施 | 实现细节 |
|------|------|---------|
| 1. 传输层 | 文件大小限制 | `MAX_CONTENT_LENGTH = 32MB`，超过返回 413 |

| 6. 输出层 | XSS 防护 | 零 `innerHTML`, 全 `textContent` + DOM API + 事件委托 |

| 8. 部署层 | 服务绑定 + 调试控制 | `host="127.0.0.1"` 仅本地, `FLASK_DEBUG` 环境变量控制 |

### 5.3 文件名净化 (`sanitize_filename`)

```python
def sanitize_filename(filename: str) -> str:
    filename = unicodedata.normalize("NFKD", filename)  # Unicode 规范化
    filename = re.sub(r"[\x00-\x1f\x7f-\x9f]", "", filename)  # 去除控制字符
    filename = re.sub(r'[<>:"/\\|?*]', "_", filename)  # 替换危险字符
    if len(filename) > 200:  # 长度限制
        name_part, ext_part = os.path.splitext(filename)
        filename = name_part[:195] + ext_part
    return filename.strip()
```

## 七、创新点与技术亮点

### 7.1 技术创新

1. **PDF 三级降级 OCR 策略**: PyPDF(文本层) → PyMuPDF(渲染层) → API OCR(多模态)，自动适配文本型/扫描型/混合型 PDF
2. **云端 OCR 方案**: 利用 DashScope qwen-vl-max 多模态 API 替代本地 OCR 引擎，避免 Python 版本兼容和大型模型下载问题，中文识别准确率高
3. **LangChain LCEL 声明式链**: `|` 管道操作符构建 RAG pipeline，参数化 `streaming` 合并重复函数，代码量减少 40%
4. **Chain 热缓存**: `_get_chain()` 复用 LLM HTTP 连接池，文件变更时自动失效重建，减少 90% 的连接开销
5. **SSE 异步生成器**: 前端 `async function* parseSSEStream()` 用 for-await-of 消费流，代码清晰可测试

### 7.2 工程创新

1. **Loader Registry 模式**: 5 大类格式通过字典注册，扩展名验证由此派生，新增格式仅需一个 key-value 对
2. **基础安全配置**: 限流/认证/CSP/XSS/文件净化/输入校验多层面覆盖，安全审计通过
3. **零 XSS 前端架构**: 从设计层面禁用 innerHTML，全量 DOM API + textContent
4. **智能错误处理**: 通用用户消息 + 详细 traceback 日志 (exc_info=True) 双通道
5. **知识库状态持久化**: JSON 快照 + 磁盘文件校验双层保障，重启后全自动恢复，用户无感知
6. **Windows 兼容性**: 路径处理、文件句柄释放、编码处理均经过 Windows 实际环境测试
