# 文档智能分析系统 — 答辩文档

## 一、项目概述

### 1.1 项目背景

在大数据时代，企业和个人面临海量文档数据（PDF 报告、Word 文档、PPT 演示、Excel 表格、扫描件、网页、技术手册等），传统的关键词搜索无法满足语义级的深度检索需求。本项目构建了一个基于 **RAG（检索增强生成）** 的文档智能分析系统，用户上传任意常见文档后，系统自动将非结构化文档转化为结构化 JSON 并建立向量知识库，支持基于文档内容的自然语言问答。

### 1.2 核心功能

| 功能 | 描述 |
|------|------|
| 多格式文档上传 | 支持 PDF(含扫描版)/Word(.doc/.docx)/PPT/Excel/HTML/图片/文本等 **5 大类格式** |
| 智能文本提取 | 根据文件类型自动选择加载器；PDF 三级降级策略(文本→PyMuPDF→OCR) |
| 扫描件 OCR | 扫描版 PDF 和图片通过 DashScope 多模态 API (qwen-vl-max) 识别文字 |
| JSON 结构化 | 智能分块(chunk)，生成带完整元数据的结构化 JSON |
| 向量知识库构建 | DashScope 嵌入 + Chroma 持久化存储，支持增量追加 |
| AI 智能问答 | 基于 RAG 的语义检索 + DeepSeek-V3 生成，SSE 流式返回 |
| **知识库持久化** | **重启后自动恢复文件列表，无需重新上传，向量数据持久化在 chroma_db/** |
| 安全防护 | 多层防护（限流/认证/CSP/XSS/文件净化/Prompt注入防御/路径安全/错误脱敏） |

## 三、核心模块详解

### 3.1 文档加载模块 (`document_loader.py`) — 5大类格式 + OCR

**设计模式:** 策略模式 (Loader Registry)

**支持格式一览:**

| 类别 | 格式 | 加载器 | 说明 |
|------|------|--------|------|
| 文档 | `.pdf` | `load_pdf` | 三级降级：PyPDF → PyMuPDF → API OCR |
| 文档 | `.doc` | `load_doc` | 四级降级：python-docx → antiword → Word COM → 原始二进制 |
| 文档 | `.docx` | `load_docx` | python-docx 段落提取 |
| 演示 | `.pptx` | `load_pptx` | python-pptx 幻灯片文本 |
| 表格 | `.xlsx` | `load_xlsx` | openpyxl 工作表数据 |
| 网页 | `.html` `.htm` | `load_html` | BeautifulSoup 去标签提取 |
| 图片 | `.png` `.jpg` `.jpeg` `.bmp` `.tiff` `.tif` | `load_image` | API OCR 识别 |
| 文本 | `.txt` `.md` `.csv` `.json` `.xml` `.log` | `load_text` | UTF-8 直接读取 |

**PDF 三级降级 OCR 策略（核心创新）:**

```
PDF 文件上传
    │
    ├── 第1步: PyPDF 提取文本层
    │   └── 有内容 ✓ → 直接返回 (最快)
    │
    ├── 第2步: PyMuPDF (fitz) 提取
    │   └── 有内容 ✓ → 返回 (兼容更多 PDF 变体)
    │
    └── 第3步: 逐页渲染为图片 → DashScope qwen-vl-max 多模态 API OCR
        └── 日志: "OCR 处理第 X/N 页..."
        └── 有内容 ✓ → 返回 (扫描版/图片型 PDF)
        └── 无内容 ✗ → 提示"文档可能为纯图片或手写内容"
```

**OCR 实现 — 云端多模态方案（避免本地依赖冲突）:**

```python
def _ocr_image(image_bytes: bytes) -> str:
    """使用 DashScope 多模态模型 (qwen-vl-max) 进行 OCR"""
    client = OpenAI(api_key=DASHSCOPE_API_KEY, base_url=BASE_URL)
    img_b64 = base64.b64encode(image_bytes).decode("utf-8")

    response = client.chat.completions.create(
        model="qwen-vl-max",
        messages=[{
            "role": "user",
            "content": [
                {"type": "image_url", "image_url": {"url": f"data:image/png;base64,{img_b64}"}},
                {"type": "text", "text": "请完整提取并输出这张图片中的所有文字内容..."},
            ],
        }],
        temperature=0.1,
        max_tokens=4096,
    )
    return response.choices[0].message.content
```

**选择 API OCR 而非本地 OCR 的原因:**
- Python 3.13 + Windows 环境下，PaddleOCR/EasyOCR 存在 NumPy 版本冲突和网络下载障碍
- DashScope 多模态 API 国内可访问，中文识别准确率高
- 零本地模型文件（~2GB），部署轻量
- PyMuPDF `try/finally` 确保文件句柄释放，解决 Windows 文件占用问题

**Loader 注册表关键设计:**
- `LOADERS` 字典是格式注册的**唯一权威源**，扩展名验证由此派生
- 新增格式只需添加一个 key-value 对，符合开闭原则
- 元数据（filename, file_type）由 `load_file` 统一注入，各 loader 不重复设置

### 3.2 文本处理模块 (`text_processor.py`)

**核心流程:**

```
原始文档 → RecursiveCharacterTextSplitter → 结构化 JSON
            chunk_size=1500, overlap=300
```

**JSON 输出结构:**

```json
[{
  "chunk_id": 0,
  "content": "人工智能(AI)是计算机科学的一个重要分支...",
  "char_count": 188,
  "metadata": {
    "source": "D:\\...\\uploads\\abc123_test.txt",
    "file_type": ".txt",
    "filename": "abc123_test.txt",
    "page": 0
  }
}]
```

**设计亮点:**
- 使用 `@dataclass(frozen=True)` 的 `ProcessResult` 替代裸 dict，类型安全 + IDE 自动补全
- `chunk_overlap=300` 保证相邻 chunk 间有足够的上下文重叠，防止关键信息在分块边界丢失
- `docs_to_json()` 使用列表推导式，简洁高效
- **17 项 pytest 测试用例，100% 代码覆盖率**

### 3.3 向量存储模块 (`vector_store.py`)

```
文本 Chunk → DashScopeEmbeddings(text-embedding-v2) → 1024维向量 → ChromaDB (持久化)
```

**关键设计:**
- `@lru_cache(maxsize=1)` 缓存嵌入模型实例，避免每次操作重建 HTTP 连接
- 持久化到 `chroma_db/` 目录，服务重启后无需重新处理文档
- 支持增量添加：`add_documents()` 向已有知识库追加新文档，无需重建

### 3.4 RAG 问答链 (`qa_chain.py` + `rerank.py`)

**架构改进：检索+重排序前置于路由层，chain 只负责 LLM 生成**

将检索和重排序从 LCEL chain 中拆出，在路由层执行 pre-flight：
- rerank 降级时可向前端推送 warning 事件（不再静默 fallback）
- 取消机制(stream_id)可以介入检索完成后、LLM 生成前

```python
def create_chain_from_docs(docs: List[Document], *, streaming: bool = False):
    """使用预检索（和预重排序）的文档创建 RAG chain。"""
    model = ChatOpenAI(api_key=DASHSCOPE_API_KEY, base_url=BASE_URL,
                       model=LLM_MODEL, streaming=streaming)
    prompt = ChatPromptTemplate.from_messages([
        ("system", SYSTEM_PROMPT),
        ("human", "{input}"),
    ])
    context_str = format_docs(docs)
    return (
        RunnablePassthrough()
        | (lambda q: {"context": context_str, "input": q})
        | prompt
        | model
        | StrOutputParser()
    )
```

**路由层 pre-flight 检索→Rerank→LLM 生成 三阶段管线:**

```
路由层 pre-flight:
    用户问题
        │
        ▼
    kb.get_retriever().invoke → 初检 12 条候选 (similarity search, k=12)
        │
        ▼
    rerank_documents() → DashScope gte-rerank (cross-encoder) 精选 Top-4
        │  ├── 成功 → RerankResult(docs, degraded=False)
        │  └── 失败 → RerankResult(docs[:4], degraded=True) → SSE warning 事件
        │
        ▼
    create_chain_from_docs(docs, streaming=True)
        │
        ▼
    ChatOpenAI(deepseek-v3.2-exp, streaming=True) → SSE 逐 token 流式输出
        │  ├── stream_id 事件 → 前端可取消 (stop按钮 / beforeunload)
        │  └── GeneratorExit → 客户端断开时停止生成
```

**Rerank 模块 (`rerank.py`) — Cross-Encoder 重排序:**

| 属性 | 实现 |
|------|------|
| 模型 | DashScope gte-rerank (cross-encoder) |
| 请求方式 | stdlib `urllib.request`，零额外依赖 |
| 返回类型 | `RerankResult(docs, degraded)` frozen dataclass |
| 降级策略 | API 超时/异常 → `RerankResult(docs[:top_k], degraded=True)` |
| 降级通知 | routes.py 检测 `degraded=True` → SSE warning 事件 → 前端黄色提示条 |
| 参数 | `top_n=4`, 候选数 ≤ top_n 时跳过重排 |

**为何加 Rerank:**
- 向量检索 (bi-encoder) 单独对 query 和 document 编码，交互约束弱
- Cross-encoder 将 `(query, doc)` 拼接后全注意力计算，相关性判断更精确
- 初检 12→精选 4，**是 RAG 检索质量提升性价比最高的单点改动**

**设计要点:**
- rerank 返回 `RerankResult` dataclass（不可变），`degraded` 字段由路由层消费
- SSE 流先发 metadata（stream_id）+ 可选 warning，再发 content
- 明确指示 LLM "找不到就说不知道"，防止幻觉
- pre-flight 分离让检索/重排/生成三个阶段可独立监控和降级

### 3.5 后端架构 — 路由/中间件/状态管理 三层分离

app.py 原先约 330 行，集路由、中间件、安全、缓存、状态持久化于一体。重构后拆分为三层：

```
app.py (薄壳, ~35行)
    │
    ├── 创建 Flask app
    ├── 注册安全中间件 (middleware.register_security_headers)
    └── 注册 API 蓝图 (routes.api)

routes.py (~120行)              # 路由层 — 仅处理 HTTP 请求/响应
    ├── POST /upload             → check_rate_limit → process_and_index_file → save_knowledge_state
    ├── POST /chat               → check_rate_limit → get_chain → SSE stream
    ├── GET /api/status          → is_ready → processed_files
    ├── GET /api/files           → processed_files
    └── DELETE /api/files/<id>   → 清理文件/JSON/向量 → save_knowledge_state

middleware.py (~80行)           # 中间件层 — 无状态安全工具
    ├── sanitize_filename()       # Unicode 净化/控制字符/危险字符/长度
    ├── check_rate_limit()        # 滑动窗口限流 (upload:10/h, chat:30/m)
    ├── require_auth()            # Bearer Token 认证装饰器 (可选)
    └── register_security_headers()  # CSP/X-Frame/X-Content/Referrer 安全头

state_manager.py (~155行)       # 状态管理层 — KnowledgeBase Class + threading.Lock
    ├── class KnowledgeBase       # 线程安全封装：_lock / _vectorstore / _processed_files
    ├── kb.init_vectorstore()     # 启动时加载 Chroma + 恢复 knowledge_state
    ├── kb.get_retriever()        # 缓存 retriever，向量库变更时自动失效
    ├── kb.process_and_index_file()  # 文档处理 → 向量入库 → 缓存失效
    ├── kb.add_file_record() / remove_file()  # 文件记录增删 (自动持久化)
    └── kb.is_ready / files / file_count  # 只读属性（短暂持锁）
```

**分层收益:**
- **可测试性**: 每层可独立 mock 测试，KnowledgeBase 类可直接实例化，不依赖 Flask test client
- **线程安全**:  保护所有共享状态，Flask debug 模式多线程也无数据竞争
- **单一职责**: 路由只管 HTTP、中间件只管安全、Class 封装状态与并发控制
- **代码导航**: 每个文件 <160 行，命名即文档，无需全文搜索

### 3.5.1 知识库持久化机制

系统重启后无需重新上传文件，所有状态自动恢复：

```
启动 python app.py
    │
    ├── 1. load_vectorstore()
    │       └── 加载 chroma_db/ → 向量数据恢复
    │
    └── 2. _load_knowledge_state()
            ├── 读取 data/knowledge_state.json
            ├── 逐条验证:
            │   ├── data/uploads/{id}_{name} 存在? + data/json/{name}.json 存在?
            │   │   ├── 都存在 → 恢复到 processed_files
            │   │   └── 缺失 → 跳过，记录日志
            └── 前端文件列表恢复正常显示
```

**持久化数据层:**

| 数据 | 存储位置 | 重启后 |
|------|---------|--------|
| 原始文件 | `data/uploads/` | 保留 |
| JSON 结构化 | `data/json/` | 保留 |
| 文件列表状态 | `data/knowledge_state.json` | 读取恢复 |
| 向量嵌入 | `chroma_db/` | 自动加载 |

**保存/加载函数 (KnowledgeBase 类方法, 调用时持锁):**

```python
def _save_knowledge_state_locked(self):
    """上传/删除后自动调用，将 _processed_files 写入 JSON。须持 _lock。"""
    with open(KNOWLEDGE_STATE_FILE, "w") as f:
        json.dump(list(self._processed_files.values()), f, ensure_ascii=False, indent=2)

def _load_knowledge_state_locked(self):
    """启动时调用，验证磁盘文件后恢复内存状态。须持 _lock。"""
    records = json.load(open(KNOWLEDGE_STATE_FILE))
    for rec in records:
        if upload_path.exists() and json_path.exists():
            self._processed_files[rec["id"]] = rec  # in-place 恢复，不 rebind
        else:
            logger.info(f"跳过已丢失的文件: {rec['filename']}")  # 自动跳过
```

| **Rerank 降级提示** | **SSE warning 事件 -> 黄色提示条** | **用户可知检索质量降级** |
| XSS 防护 | 零 `innerHTML`，全 `textContent` + DOM API | 从架构层面杜绝注入 |
| Prompt 注入防御 | 结构分隔符 + 显式防注入规则 + 文档指令隔离 | 抵抗文档/用户双重注入攻击 |
| 事件委托 | `fileList.addEventListener` + `data-file-id` | 单个监听器管理所有删除按钮 |
| DOM 更新守卫 | `_lastReadyState` 比较 | 避免无变化时的 DOM 重建 |
| 并行请求 | `Promise.all([refreshStatus(), refreshFileList()])` | 减少页面就绪等待时间 |

## 四、数据流详解

### 4.1 文档上传处理流

```
用户选择/拖拽文件
    │
    ▼
前端验证 → 扩展名白名单
    │
    ▼
Flask 接收 → 文件大小检查 (32MB MAX_CONTENT_LENGTH)
    │         → IP 频率限制 (10次/小时)
    │         → 可选 Bearer Token 认证
    ▼
文件名净化 → sanitize_filename()
    │         Unicode 规范化 + 控制字符移除 + 特殊字符替换 + 200字符截断
    ▼
保存到 data/uploads/{file_id}_{sanitized_name}
    │
    ▼
document_loader.load_file()
    ├── .pdf  → PyPDF → PyMuPDF → API OCR (三级降级)
    ├── .doc  → 四级降级（python-docx → antiword → Word COM → 原始提取）
    ├── .docx → python-docx 段落提取
    ├── .pptx → python-pptx 幻灯片文本
    ├── .xlsx → openpyxl 工作表数据
    ├── .html/.htm → BeautifulSoup 去标签
    ├── .png/.jpg/... → API OCR 识别
    └── .txt/.md/.csv/... → 直接读取
    │
    ▼
text_processor.process_documents()
    ├── RecursiveCharacterTextSplitter(chunk_size=1500, overlap=300)
    ├── docs_to_json() → 结构化 JSON (chunk_id + content + char_count + metadata)
    └── save_json() → data/json/{filename}.json
    │
    ▼
空内容检查 → chunk_count == 0? → 提示"文档内容为空或扫描件无法识别"
    │
    ▼
vector_store.create_vectorstore() 或 add_documents()
    ├── DashScopeEmbeddings(text-embedding-v2) → 1024维向量
    └── Chroma.from_documents() → 持久化到 chroma_db/
    │
    ▼
_invalidate_cache() → 清除缓存的 RAG chain (下次请求重建)
    │
    ▼
_save_knowledge_state() → 文件列表持久化到 data/knowledge_state.json
    │
    ▼
返回 JSON: {success, file_id, filename, chunks, total_chars, json_path}
```

### 4.2 问答检索流

```
用户输入问题
    │
    ▼
Flask 接收 → 非空检查 → 长度检查 (≤2000字符)
    │         → IP 频率限制 (30次/分钟)
    │         → 知识库就绪检查 (kb.get_retriever() is not None)
    ▼
Phase 1 — Pre-flight 检索 + Rerank (同步, 快速)
    ├── kb.get_retriever().invoke(question) → 初检 12 条候选
    ├── rerank_documents(query, docs, top_k=4) → RerankResult
    └── degraded=True? → SSE 推送 warning 事件
    │
    ▼
Phase 2 — LLM 流式生成
    ├── create_chain_from_docs(rerank_result.docs, streaming=True)
    ├── chain.stream(question) → 逐 token 产出
    ├── 每 token 间检查 cancel_event.is_set()
    ├── GeneratorExit → 客户端断开时停止生成
    └── SSE 格式化 → "data: {json}\n\n" → Flask Response (text/event-stream)
    │
    ▼
前端 parseSSEStream():
    ├── data.stream_id → currentStreamId (可取消)
    ├── data.warning → addWarningMessage() 黄色提示
    ├── data.content → contentEl.textContent += chunk 逐字渲染
    └── data.done → 完成, 消息存入 localStorage
```

## 五、安全设计

### 5.1 基础安全配置

| 层级 | 措施 | 实现细节 |
|------|------|---------|
| 1. 传输层 | 文件大小限制 | `MAX_CONTENT_LENGTH = 32MB`，超过返回 413 |
| 2. 应用层 | 频率限制 | upload: 10次/小时, chat: 30次/分钟, 429 响应 |
| 3. 认证层 | API Token (可选) | `require_auth` 装饰器, Bearer Token, 未配置则跳过 |
| 4. 头部层 | 安全响应头 | HSTS / CSP / X-Frame-Options DENY / X-Content-Type nosniff / Permissions-Policy |
| 5. 输入层 | 文件名净化 + 问题长度限制 | `sanitize_filename()` + `MAX_QUESTION_LENGTH=2000` |
| 6. 输出层 | XSS 防护 | 零 `innerHTML`, 全 `textContent` + DOM API + 事件委托 |
| 7. 日志层 | 信息脱敏 + 分级异常 | `exc_info=True` 记录详细 traceback；`DocumentLoaderError` 子类返回友好中文提示；区分 400(用户错误)/500(服务端错误)；不再出现 Flask 默认 500 HTML 页面 |
| 8. 部署层 | 服务绑定 + 调试控制 | `host="127.0.0.1"` 仅本地, `FLASK_DEBUG` 环境变量控制 |

### 5.2 密钥管理

```
.env (gitignored)
  → config.py: os.environ.get("DASHSCOPE_API_KEY")
  → 启动时 config.py RuntimeError 校验 (key 不存在则拒绝启动)
  → 运行时各模块通过 config.DASHSCOPE_API_KEY 获取
```

- API Key **从不硬编码**在源码中
- `.env` 被 `.gitignore` 排除，防止泄露到版本控制
- 限流字典空键清理：`del _requests_store[key]` 防止长期运行内存泄漏

### 5.3 文件名净化 (`sanitize_filename`)

```python
def sanitize_filename(filename: str) -> str:
    filename = unicodedata.normalize("NFKD", filename)  # Unicode 规范化
    filename = re.sub(r"[\x00-\x1f\x7f-\x9f]", "", filename)  # 去除控制字符
    filename = re.sub(r'[<>:"/\\|?*]', "_", filename)  # 替换危险字符
    if len(filename) > 200:  # 长度限制
        name_part, ext_part = os.path.splitext(filename)
        filename = name_part[:195] + ext_part
    return filename.strip()
```

## 七、创新点与技术亮点

### 7.1 技术创新

1. **PDF 三级降级 OCR 策略**: PyPDF(文本层) → PyMuPDF(渲染层) → API OCR(多模态)，自动适配文本型/扫描型/混合型 PDF
2. **Cross-Encoder Rerank + 降级通知**: 初检 12→精选 4，DashScope gte-rerank 做 token-level 语义评分；RerankResult.degraded 标记 + SSE warning 事件实现用户可感知降级
3. **云端 OCR 方案**: 利用 DashScope qwen-vl-max 多模态 API 替代本地 OCR 引擎，避免 Python 版本兼容和大型模型下载问题，中文识别准确率高
4. **Pre-flight 检索+重排架构**: 路由层先同步完成检索和重排序，再流式 LLM 生成；三个阶段独立可监控、可降级
5. **SSE 流式取消机制**: stream_id + threading.Event cancel 信号 + 前端 navigator.sendBeacon，用户离开页面后停止 token 消耗
6. **线程安全 Class 封装**: KnowledgeBase 类 + threading.Lock，解决 Python 模块级全局变量的 import 引用陷阱和并发竞争问题

### 7.2 工程创新

1. **Loader Registry 模式**: 5 大类格式通过字典注册，扩展名验证由此派生，新增格式仅需一个 key-value 对
2. **路由/中间件/Class 状态管理三层分离**: app.py 从 330 行拆分为 35 行薄壳 + routes + middleware + KnowledgeBase Class，线程安全、可独立测试
3. **自定义异常分级**: DocumentLoaderError -> UnsupportedFileTypeError / EmptyDocumentError / DocumentParseError，前端精准展示中文错误提示，不再出现 Flask 500 HTML
4. **零 XSS 前端架构**: 从设计层面禁用 innerHTML，全量 DOM API + textContent
5. **智能错误处理**: 通用用户消息 + 详细 traceback 日志 (exc_info=True) 双通道
6. **知识库状态持久化**: JSON 快照 + 磁盘文件校验双层保障 + in-place mutation 避免引用陷阱，重启后全自动恢复

8. **TDD 测试驱动**: 46 项 pytest 用例 (text_processor 17 + state_manager 15 + rerank 7 + document_loader 7)，frozen dataclass + 线程安全并发测试

## 十、本地模型部署指南

系统架构天然支持替换为本地模型。LLM、Embedding、Rerank、OCR 四个组件均通过 **OpenAI 兼容 API** 调用，任何兼容服务都可通过修改 `config.py`（或 `.env`）接入。

### 10.1 架构原理

```
config.py 中的配置 (可通过 .env 覆盖)
    │
    ├── LLM_MODEL        → 默认 deepseek-v3.2-exp → 可改为本地模型名
    ├── EMBEDDING_MODEL  → 默认 text-embedding-v2 → 可改为本地嵌入模型
    ├── BASE_URL         → 默认 DashScope API     → 可改为本地服务地址
    └── DASHSCOPE_API_KEY → 默认 DashScope Key    → 可改为本地服务 token
```

所有组件通过 `ChatOpenAI` / `OpenAIEmbeddings` 调用，这两个类只依赖 `base_url` + `api_key` + `model`，不绑定任何特定厂商。

### 10.2 方案一：Ollama（推荐，最简单）

**Ollama** 是最易用的本地模型运行工具，一键安装，模型下载后自动管理。

#### 安装 Ollama

```bash
# Windows: 下载安装包 https://ollama.com/download/windows
# Linux:
curl -fsSL https://ollama.com/install.sh | sh
# macOS:
brew install ollama
```

#### 拉取所需模型

```bash
# LLM — 中文问答（推荐 qwen2.5，中文能力优秀）
ollama pull qwen2.5:14b        # 14B 参数，需要 8GB+ 显存
ollama pull qwen2.5:7b         # 7B 参数，需要 4GB+ 显存（轻量选择）
ollama pull deepseek-r1:8b     # 推理能力强，需要 5GB+ 显存

# Embedding — 文本向量化
ollama pull nomic-embed-text   # 768 维，轻量高效
ollama pull bge-m3             # 1024 维，多语言，需要 2GB+ 显存
```

#### 配置系统接入 Ollama

在 `.env` 文件中添加（或直接修改 `config.py`）：

```bash
# .env — 使用 Ollama 本地模型
DASHSCOPE_API_KEY=ollama           # Ollama 不需要真实 key，但不能为空
LLM_MODEL=qwen2.5:14b              # 改成你拉取的模型名
EMBEDDING_MODEL=nomic-embed-text   # 改成你拉取的嵌入模型名
BASE_URL=http://localhost:11434/v1 # Ollama 默认端口 11434，OpenAI 兼容路径 /v1
```

#### 启动

```bash
# 1. 确保 Ollama 在运行（Windows 下启动桌面应用即可）
ollama serve

# 2. 启动问答系统
python app.py
# → http://localhost:5000
```

**注意事项：**
- Ollama 的 Rerank 支持有限，需在 `config.py` 中将 `rerank.py` 的 rerank 调用做降级处理（代码已内置降级逻辑，rerank API 不可用时自动 fallback 到向量排序）
- OCR 功能（扫描版 PDF/图片）在纯本地模式下需要额外处理——Ollama 不支持多模态 OCR，建议保留 DashScope OCR 或使用 llava 等视觉模型替代

### 10.4 方案三：vLLM / LocalAI / LM Studio

任何提供 OpenAI 兼容 `/v1/chat/completions` 和 `/v1/embeddings` 端点的服务都可接入：

| 服务 | 默认地址 | 特点 |
